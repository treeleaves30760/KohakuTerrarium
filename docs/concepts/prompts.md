# Prompt System

How KohakuTerrarium builds system prompts with configurable tool lists, sub-agent lists, framework hints, and terrarium-specific channel awareness.

## Overview

The system prompt is assembled from multiple sources in a specific order. The agent personality comes from the authored prompt file. Everything else (tool list, syntax hints, channel topology) is auto-generated and injected.

```
Final System Prompt
  = Base prompt (from system.md)
  + Tool/sub-agent list (auto-generated from registry)
  + Framework hints (tool call syntax, commands)
  + Channel topology (terrarium mode only)
```

The authored prompt only covers personality, methodology, and safety. Tool documentation, call syntax, and team structure are never written by hand.

## Prompt Aggregation

The core function is `aggregate_system_prompt()` in `prompt/aggregator.py`. It takes the base prompt and registry, then generates additional sections based on configuration.

### Sections (in order)

| Section | Source | Always included? |
|---------|--------|-----------------|
| Base prompt | `system.md` file (supports Jinja2 templates) | Yes |
| Tool list | Auto-generated from registry | If `include_tools=True` |
| Sub-agent list | Auto-generated from registry | If sub-agents registered |
| Channel hints | Generated if send_message/wait_channel exist | Only with channel tools |
| Output format | Explains `[/output_name]` syntax | Only with named outputs |
| Framework hints | Tool call syntax + commands | If `include_hints=True` |
| Channel topology | Team channels, workflow, rules | Terrarium mode only |

### skill_mode: Dynamic vs Static

The `skill_mode` setting controls how much tool documentation goes into the prompt.

**Dynamic mode** (default):
- Tool list shows name + one-line description only
- Full docs loaded on demand via the `info` tool
- Framework commands (`info`, `jobs`, `wait`) explained in prompt
- Lower token usage, suitable for large tool catalogs

**Static mode**:
- Full documentation for every tool included in the prompt
- No `info` command needed (everything is already there)
- Higher token usage, but model has complete context immediately

```yaml
# Agent config
controller:
  skill_mode: dynamic   # or "static"
```

### tool_format: How LLMs Call Tools

The `tool_format` setting determines what syntax examples appear in the prompt. See [Tool Formats](tool-formats.md) for the full format reference.

**bracket** (default):
```
[/tool_name]
@@param=value
content
[tool_name/]
```

**xml**:
```
<tool_name param="value">content</tool_name>
```

**native**: No syntax examples. The LLM API handles function calling natively. Tool schemas are passed via the API, not the prompt.

```yaml
controller:
  tool_format: native   # recommended for OpenAI/Anthropic models
```

### What Goes in the Base Prompt (system.md)

Only personality and methodology. Everything else is auto-generated.

| Content | Where it belongs |
|---------|-----------------|
| Agent personality/role | `system.md` |
| Agent-specific guidelines | `system.md` |
| Tool list (name + desc) | AUTO-GENERATED |
| Tool call syntax | AUTO-GENERATED (from tool_format) |
| Full tool documentation | Via `info` command (dynamic) or AUTO-GENERATED (static) |
| Channel topology | AUTO-GENERATED by terrarium runtime |
| Output format hints | AUTO-GENERATED (from named_outputs config) |

## Terrarium Mode: Channel Topology Injection

When a creature runs inside a terrarium, the runtime injects a "Team Communication" section into its system prompt. This happens via `inject_prompt_section()` after the base prompt is aggregated.

### What Gets Injected (creatures)

`build_channel_topology_prompt()` generates:

**1. Auto-Listening section**
Explains that channel messages arrive automatically as trigger events:
```
[Channel 'tasks' from coordinator]: Please implement the auth fix
[Channel 'team_chat' (broadcast) from reviewer]: Status update...
```
Includes the critical note: "Hearing a message does NOT mean you must respond."

**2. Critical Rules**
- All output to other creatures MUST use `send_message`
- Your text output is invisible to other creatures
- Sub-agents are internal tools, NOT team members

**3. Workflow**
Based on the creature's `listen` and `send` channels:
```
1. You receive tasks/messages on: `tasks`, `feedback`
2. Do your work using your tools and sub-agents
3. Send your results via `send_message` to: `results`, `review`
```

**4. Channel Listing**
Each channel with its type (queue/broadcast) and the creature's role (listen/send):
```
- `tasks` [queue] (listen) -- Work assignments from coordinator
- `results` [queue] (send) -- Completed work for review
- `team_chat` [broadcast] (listen, send) -- Team-wide updates
- `swe` [queue] (listen) -- your direct channel
```

**5. Team Members**
Lists sibling creatures for direct messaging.

### What Gets Injected (root agent)

`build_root_awareness_prompt()` generates a different section for the root:

- Root auto-listens to ALL channels (not just assigned ones)
- Guidance on what to do with each channel type:
  - Results: summarize for user
  - Broadcast: absorb context, only act if relevant
  - Tasks: usually handled by creatures, not root
- Full creature and channel listings
- Direct channel documentation

### Key Distinction: Creature vs Root

| Aspect | Creature | Root |
|--------|----------|------|
| Listens to | Assigned channels only | ALL channels |
| Prompt section | `build_channel_topology_prompt` | `build_root_awareness_prompt` |
| Workflow | Receive -> work -> send results | Delegate -> wait -> summarize |
| "Must respond?" | Queue messages: yes. Broadcast: only if relevant | Summarize results. Broadcast: absorb. Tasks: ignore |

## Plugin System (Advanced)

For advanced prompt customization, the plugin architecture (`prompt/plugins.py`) provides modular composition with priority ordering.

### Built-in Plugins

| Plugin | Priority | What it adds |
|--------|----------|-------------|
| `EnvInfoPlugin` | 10 | Working directory, git status, platform, date |
| `ProjectInstructionsPlugin` | 20 | AGENTS.md, .kohaku.md, or CLAUDE.md content |
| `ToolListPlugin` | 50 | Tool names + descriptions |
| `FrameworkHintsPlugin` | 60 | Tool call syntax, execution model |

Lower priority = earlier in prompt. Use `aggregate_with_plugins()` instead of `aggregate_system_prompt()` to use plugins.

### Default Plugin Sets

```python
from kohakuterrarium.prompt.plugins import get_default_plugins, get_swe_plugins

# Basic: tools + hints
plugins = get_default_plugins()

# SWE: env info + project instructions + tools + hints
plugins = get_swe_plugins()
```

## Prompt Inheritance via base_config

When a creature inherits from another via `base_config`, system prompts are concatenated:

1. Load base creature's `system.md`
2. Load child creature's `system.md`
3. Concatenate with double newline

For `creatures/swe` inheriting from `creatures/general`:
```
[general/prompts/system.md content]

[swe/prompts/system.md content]
```

The tool list and hints are generated from the merged config (child's tools extend base's tools).

## Configuration Reference

### Agent Config (YAML)

```yaml
controller:
  skill_mode: dynamic         # "dynamic" or "static"
  tool_format: native         # "bracket", "xml", or "native"
  include_tools_in_prompt: true
  include_hints_in_prompt: true

system_prompt_file: prompts/system.md

# Named outputs (triggers output format hints)
output:
  named_outputs:
    discord: { type: custom, ... }
```

### Terrarium Config (YAML)

```yaml
terrarium:
  creatures:
    - name: swe
      channels:
        listen: [tasks, feedback]    # Auto-injected into topology prompt
        can_send: [results, review]  # Auto-injected into topology prompt

  channels:
    tasks:
      type: queue
      description: "Work assignments"    # Shown in topology prompt
    team_chat:
      type: broadcast
      description: "Team-wide updates"   # Shown in topology prompt
```
