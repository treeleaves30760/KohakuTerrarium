# 什么是 Agent

在 KohakuTerrarium 的语汇里，一个 Agent 就是一只 **Creature (creature)**。要了解为什么Creature长成现在这个样子，从零开始推一次比较快。

## Stage 0 — 聊天机器人

```
  +-------+  +-----+  +--------+
  | Input | ---> | LLM | ---> | Output |
  +-------+  +-----+  +--------+
```

用户输入、模型回答、打印答案。这是对话式 AI 最简单的样子。它能 **说，** 但不能 ** 做**。

## Stage 1 — 加上工具系统

```
  +--------------+
  | Tools System |
  +------+-------+
  ^  |
  |  v
  +-------+  +---------+  +--------+
  | Input | ---> |  LLM  | ---> | Output |
  +-------+  +---------+  +--------+
```

让 LLM 有能力调用工具：shell 指令、文件编辑、网页搜寻。这样它可以 **做事** 了。它不再是聊天机器人，而是 smolAgent 或 swe-Agent 意义下的 Agent。

这已经有用了。但它也已经有限制：要让这个 Agent 做任何事，唯一的方法就是 **对它打字**。

## Stage 2 — 加上触发器系统

```
  +--------------+
  +--> | Tools System |
  |  +--------------+
  |  ^  |
  |  |  v
  +-------+  +---------+v  +---------+  +--------+
  | Input |-->| Trigger |----->|  LLM  | ---> | Output |
  +-------+  | System  |  +---------+  +--------+
  +---------+
```

真实的 Agent 需要不需要用户输入也能把自己叫醒：`/loop` 功能、背景 job 完成、idle check、webhook、计时器。某个东西必须盯着世界、在条件满足时启动控制器。那个东西叫 **触发器 (trigger)**。

一旦你有了触发器，你会发现 **用户输入本身不过是另一种触发器**。所以「输入」变成一个更泛化的唤醒机制的特例。Claude Code 和 OpenClaw 停在这一阶段。

## Stage 3 — 加上子 Agent

```
  +--------------+
  +--> | Tools System |
  |  +--------------+
  |  ^  |
  |  |  v
  +-------+  +---------+v  +---------+  +--------+
  | Input |-->| Trigger |----->|  LLM  | ---> | Output |
  +-------+  | System  |  +---------+  +--------+
  +---------+  ^  |
  |  v
  +--------------+
  |  Sub Agents  |
  +--------------+
```

上下文视窗有限。有很多探索性子步骤的长任务，如果每一步都塞到父代理的对话里，预算一下就爆了。

解决办法是生一个 **子 Agent (sub-agent)** ：独立上下文的嵌套Creature，跑完之后回报一个浓缩结果、然后消失。重点是：从父代理的角度看，子 Agent** 也是一个工具**— 你叫它、它返回。现代 coding Agent 都停在这一阶段。

## 六模块的 Creature

拼起来：

| 模块 | 角色 |
|---|---|
|**控制器 (Controller)**| 推理回圈。接 LLM 串流、解析工具调用、派发。 |
|**输入 (Input)**| 告诉控制器现在要干嘛 (触发器的一种特例)。 |
|**触发器 (Trigger)**| 当外界需要时启动控制器。 |
|**工具 (Tool)**| Agent 用来做事情的东西。 |
|**子 Agent (Sub-Agent)**| 嵌套Creature — 概念上也是一种工具。 |
|**输出 (Output)**| Agent 向外界说话的方式。 |**Creature (creature)**就是这六个模块的集合体。这是 KohakuTerrarium 的第一等抽象，后面每一份使用指南、每一份参考、每一份其他的概念文件，最终都是在讲这件事。

有一件事值得一开始就厘清：当你在根 README 或指南里看到「五个可扩充的模块类型」这类说法时，指的是用户可以换掉实现的那五个 — 输入、触发器、输出、工具、子 Agent。第六个模块 — 控制器 — 是框架提供的推理回圈，你可以 **配置** 它 (LLM、skill mode、tool format) 但不会整个换掉实现。同样的六个模块，只是补一句谁实现哪一个。

## 不要被这个推导绑死

上面这个故事是一个好用的预设，但不是铁律。实务上：

- 只有触发器的 Creature可以完全省略输入 (`input: none`)。排程 job 型的 Agent 不需要用户。
- 没有输出的 Creature是合法的 (只做 side-effect)。
- 没有工具的 Creature可以存在 (纯回复型 Agent)。
- 框架会在方便时弯曲自己的抽象：Terrarium的频道技术上是「某个Creature的工具写入；另一个Creature的触发器触发」这个 pattern，它不清晰地属于任何一个模块。没关系。参见 [边界](../boundaries.md)。
